# 传票系统设计书

## 1. 系统概述

### 1.1 背景

工厂制造工序流转管理系统。通过水墨屏（电子墨水屏）实时显示 lot 当前工序及基础数据，配合完工录入和不良录入，实现工序全流程追踪。

### 1.2 功能模块

| 模块 | 类型 | 说明 |
|------|------|------|
| 水墨屏绑定页面 | 前端页面 | 将水墨屏与 lot 绑定，计算工序序列，推送显示数据 |
| 完了输机页面 | 前端页面 | 各工序完工数据录入（目前仅 gp/磁石吸取/清空） |
| 不良入力页面 | 前端页面 | 当前工序不良数据录入与查看 |
| 工序排列服务 | 后端服务 | 固定工序 + 额外工序合并排序，绑定时触发 |
| 工序流转服务 | 后端服务 | 根据感应器进出信号判定工序切换，刷新水墨屏 |

### 1.3 技术栈

- 后端: Java 17 + Spring Boot 2.5.15 + MyBatis
- 前端: Vue 2 + Element UI
- 数据库: MySQL（使用现有 ry_user 库）
- 通信: WebSocket（接收感应器数据）

---

## 2. 数据库设计

所有表建在 **ry_user** 库中，表名前缀 `ticket_`。

> **设计原则**：甲方已有的数据（lot 基础数据、固定工序、额外工序）不建中间表，绑定时通过模块化接口直接读取甲方的表。我们只建自己系统产生和维护的数据表。

### 2.1 process_code_mapping — 工序 code 映射表

工序 code 与中文名的字典表，全系统通用。

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| process_code | varchar(20) | PK | 工序 code |
| process_name | varchar(50) | Y | 工序中文名 |

> 纯字典，两个字段。流转引擎拿到 code 后查此表翻译成中文名。
>
> **测定回数条目**（必须预置）：
>
> | process_code | process_name |
> |---|---|
> | TEST_01 | 一回测定 |
> | TEST_02 | 二回测定 |
> | TEST_03 | 三回测定 |
> | ... | ... |
> | TEST_10 | 十回测定 |
>
> 流转引擎根据 `testing_round` 拼出 code：`TEST_` + 补零的回数 → 查此表得到中文名。
> sensor_process_mapping 中测定棚架的感应器统一映射到 `TEST_01`，流转引擎内部根据回数替换为实际 code。


## 3. 模块化数据接口

甲方的数据源结构未定，以下三个接口封装了对甲方数据的读取。等甲方给了表结构，各写一个实现类即可，系统其余部分完全不用动。

### 3.1 lot 基础数据接口

```java
public interface LotDataProvider {

    /**
     * 读取 lot 基础数据（品名、品番、客户等）
     * 数据来源：甲方的表，结构待定
     */
    LotBasicInfo getLotInfo(String lotNo);
}
```

返回值 `LotBasicInfo` 包含：lotNo、productName、productModel、customer 等字段。

### 3.2 固定工序接口

```java
public interface ProcessDataProvider {

    /**
     * 读取某 lot 的固定工序列表（带顺序）
     * 数据来源：甲方提供的表，结构待定
     */
    List<ProcessStep> getFixedProcesses(String lotNo);

    /**
     * 读取某品名的额外工序列表
     * 数据来源：甲方提供的另一张表，结构待定
     */
    List<ProcessStep> getExtraProcesses(String productName);
}
```

返回值 `ProcessStep` 包含：processCode、processName、seq。

### 3.3 使用方式

```
绑定时:
  1. LotDataProvider.getLotInfo(lotNo)         → 拿到基础数据
  2. ProcessDataProvider.getFixedProcesses(lotNo)   → 拿到固定工序
  3. ProcessDataProvider.getExtraProcesses(productName) → 拿到额外工序
  4. 合并 → 写入 lot_process_sequence
  5. 组装显示数据 → 写入 eink_display_content

流转时:
  1. LotDataProvider.getLotInfo(lotNo)         → 重新拿基础数据
  2. 从 lot_process_sequence 读下一工序
  3. 组装显示数据 → 更新 eink_display_content
```

> 甲方给了表结构后，写实现类读他的表就行。换数据源 = 换实现类，其他代码零改动。

## 4. 接口设计

### 4.1 WebSocket 接口

#### WS /ws/sensor — 接收感应器数据

**消息格式**（目前已知的绑定页面格式，后续其他格式通过适配器统一）：

```json
{
  "rfid": "E200001234567890",
  "ant": "1",
  "ip": "192.168.1.100",
  "status": 1,
  "remark": "",
  "time": "2026-02-10 10:30:00"
}
```

**处理逻辑**：

1. 记录到 `sensor_event_log`（纯日志，不用于业务查询）
2. 根据 `ip` 查 `sensor_process_mapping`：
   - **bind 类型感应器**：推送 `{ rfid, processCode, processName }` 到前端绑定页面
   - **flow 类型感应器**：根据 rfid 查 `rfid_lot_binding` 拿到 lot_no → 根据 ip 查 `sensor_process_mapping` 拿到 processCode → 构建 `UnifiedFlowEvent { lotNo, processCode, eventType(IN/OUT) }` → 送入流转引擎

---

### 4.2 REST API — 水墨屏绑定

#### POST /api/ticket/eink/bind

**请求**：

```json
{
  "rfid": "E200001234567890",
  "lotNo": "LOT20260210001",
  "processCode": "2350"
}
```

> rfid 和 processCode 由 WS 推送到前端，lotNo 由用户手动输入。processCode 是业务概念，与底层数据源无关。

**处理流程**：

```
1. 删除 rfid_lot_binding 中该 rfid 的旧绑定（status 置为 1）
2. 调用 LotDataProvider.getLotInfo(lotNo) 获取基础数据（品名、品番、客户等）
3. 查询 process_code_mapping 获取工序中文名
4. 调用【工序排列服务】→ 生成 lot_process_sequence
5. 查询 completion_detail（lot_no + gp工序code）获取 gp_defect_info
6. 插入/更新 eink_display_content
7. 插入 rfid_lot_binding（新绑定）
8. 初始化 lot_flow_status（当前工序、has_exited=0、testing_round=0）
9. 调用水墨屏 API 推送显示数据（预留接口）
```

**响应**：

```json
{
  "code": 200,
  "msg": "绑定成功",
  "data": {
    "lotNo": "LOT20260210001",
    "processName": "dh处理",
    "productName": "XXX"
  }
}
```

## 6. 核心服务设计

### 6.1 工序排列服务

**触发时机**：水墨屏绑定提交时调用。

**输入**：lot_no

**输出**：排好序的工序列表，写入 `lot_process_sequence`

**逻辑**：

```
步骤1: 调用 LotDataProvider.getLotInfo(lotNo) 获取品名（productName）
步骤2: 调用 ProcessDataProvider.getFixedProcesses(lotNo) 获取固定工序列表
        → 结果: [2350, 2400, 3300, 3500, 3410, 3600, 4400, 4100]
步骤3: 调用 ProcessDataProvider.getExtraProcesses(productName) 获取额外工序
        → 结果: [2450, 2500, 3550, 3650]
步骤4: 合并
步骤5: 写入 lot_process_sequence
```

**步骤4 合并算法详解**：

```
fixed = [2350, 2400, 3300, 3500, 3410, 3600, 4400, 4100]  （顺序不可变）
extra = [2450, 2500, 3550, 3650]

对于每个 extraCode:
  遍历 fixed 的相邻对 (fixed[i], fixed[i+1]):
    如果 fixed[i] < extraCode < fixed[i+1]，则插入到 fixed[i] 之后
  如果 extraCode < fixed[0]，插入到最前面
  如果 extraCode > fixed[last]，插入到最后面

示例执行过程:
  2450: 2400 < 2450 < 3300 ✓ → 插入 2400 后面
  2500: 2450 < 2500 < 3300 ✓ → 插入 2450 后面（注意：此时 2450 已在列表中）
  3550: 3500 < 3550 < 3410？ 不满足（3550 > 3410）
         3410 < 3550 < 3600 ✓ → 插入 3410 后面
  3650: 3600 < 3650 < 4400 ✓ → 插入 3600 后面

最终结果:
  [2350, 2400, 2450, 2500, 3300, 3500, 3410, 3550, 3600, 3650, 4400, 4100]
```

> **注意**：这不是全局排序。固定工序的顺序绝对不变（3410 就是在 3500 后面），额外工序仅在相邻固定工序的 code 数值区间内插入。

### 6.2 工序流转服务

#### 6.2.1 模块化架构

```
    ┌─────────────────────────────────────────────────┐
    │               事件源层 (EventSource)              │
    │                                                 │
    │  ┌──────────┐  ┌──────────┐  ┌──────────┐      │
    │  │ WS感应器  │  │ 扫码枪    │  │ 手动录入  │ ... │
    │  │ Adapter  │  │ Adapter  │  │ Adapter  │      │
    │  └────┬─────┘  └────┬─────┘  └────┬─────┘      │
    │       │              │              │            │
    └───────┼──────────────┼──────────────┼────────────┘
            │              │              │
            ▼              ▼              ▼
    ┌─────────────────────────────────────────────────┐
    │          统一事件 (UnifiedFlowEvent)              │
    │                                                 │
    │  { lotNo, processCode, eventType(IN/OUT),       │
    │    timestamp, source, rawData }                  │
    └────────────────────┬────────────────────────────┘
                         │
                         ▼
    ┌─────────────────────────────────────────────────┐
    │             流转引擎 (FlowEngine)                │
    │                                                 │
    │  1. 查 lot_flow_status 获取当前状态               │
    │  2. 查 lot_process_sequence 获取工序序列          │
    │  3. 调用 FlowStrategy.evaluate() 判定            │
    │  4. 更新 lot_flow_status                         │
    │  5. 如需流转 → 触发 DisplayService               │
    └────────────────────┬────────────────────────────┘
                         │
                         ▼
    ┌─────────────────────────────────────────────────┐
    │           显示服务 (DisplayService)               │
    │                                                 │
    │  1. 调用 LotDataProvider.getLotInfo() 重新查基础数据│
    │  2. 查询 gp 不良指示                              │
    │  3. 组装显示内容                                  │
    │  4. 更新 eink_display_content                    │
    │  5. 调用水墨屏 API 推送（预留接口）                │
    └─────────────────────────────────────────────────┘
```

**扩展方式**：新增事件源时，只需实现一个 Adapter，将原始数据转为 `UnifiedFlowEvent`，其余逻辑完全复用。

#### 6.2.2 流转状态机

**状态定义**（lot_flow_status.has_exited）：

| 当前状态 | 含义 |
|---------|------|
| has_exited = 0 | lot 在当前工序中（未离开） |
| has_exited = 1 | lot 已离开当前工序（等待进入下一工序） |

**"合法的下一工序"判定**：

普通工序是线性的，下一工序只有一个（lot_process_sequence 中 seq+1）。
但测定区域存在分支，下一工序可能有多个：

```
当前工序          合法的下一工序
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
普通工序           sequence 中 seq+1（唯一）
测定(N回)          后始末 或 mips（分支：继续循环 or 结束测定）
后始末             真空热处理 或 mips（分支：继续循环 or 结束测定）
真空热处理         测定(N+1回) 或 sequence 中测定区域之后的工序（分支）
```

**事件处理规则**：

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
状态              事件                  动作
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
在工序A(未离开)    OUT(A)               → 标记 has_exited=1
在工序A(未离开)    IN(A)                → 忽略（噪音：人员重复进入）
在工序A(未离开)    OUT(非A)             → 忽略（无关事件）
在工序A(未离开)    IN(B) B是合法下一工序  → 异常！未出A就进B
                                         清空显示数据，lot显示异常
在工序A(未离开)    IN(C) C不合法        → 忽略（无关事件）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
在工序A(已离开)    IN(B) B是合法下一工序  → ✅ 流转到B！刷新水墨屏
在工序A(已离开)    IN(A)                → 回退为未离开（人员走回来了）
在工序A(已离开)    OUT(A)               → 忽略（重复出信号）
在工序A(已离开)    IN(C) C不合法        → 异常！跳工序
在工序A(已离开)    OUT(非A)             → 忽略
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

#### 6.2.3 测定回数特殊逻辑

**背景**：物理上只有一个测定棚架，lot 可能多次进出。系统通过计数器（testing_round）跟踪回数。

**测定循环路径**：

```
... → 电镀后热处理 → ┌→ 测定(N回) → 后始末 → 真空热处理 ─┐
                     │                                  │
                     └──────────────────────────────────┘
                     │
                     └→ 测定(N回) → mips → g2外选 → ...
                        (不再循环，继续后续工序)
```

**判定规则**：

```
1. 首次进入测定棚架:
   testing_round = 1, 显示「一回测定」

2. 测定出 → 后始末进 → 后始末出 → 真空热处理进 → 真空热处理出 → 测定再进:
   testing_round++, 显示「N回测定」(N = testing_round)

3. 测定出 → mips进 或其他非循环工序进:
   testing_round 归零, 正常流转

4. 最大回数: 10次
```

**实现方式**：

- `sensor_process_mapping` 中测定棚架的所有感应器统一映射到 `TEST_01`
- 流转引擎收到测定棚架的 IN 事件时：
  - 检查 `lot_flow_status.testing_round`
  - 如果来自真空热处理（即上一工序是真空热处理）→ `testing_round + 1`
  - 如果是首次进入测定 → `testing_round = 1`
  - 拼出实际 code：`TEST_` + 补零的 testing_round（如 `TEST_02`）
  - 查 `process_code_mapping` 获取中文名（如「二回测定」）
  - 用实际 code 和中文名更新 lot_flow_status 和 eink_display_content

#### 6.2.4 异常处理

| 异常场景 | 处理方式 |
|---------|---------|
| 未出当前工序就进下一工序 | lot_flow_status.status = ABNORMAL，水墨屏除 lot 外全部清空，lot 字段显示异常信息 |
| 跳工序（A→C，跳过B） | 同上 |
| 同一工序反复进出 | 忽略噪音信号，不改变状态 |
| 测定超过10次 | 记录异常日志，不再递增 |

> 异常信息的具体文案待与甲方确认，系统预留可配置的消息模板。

## 9. 模块化设计总结

本系统有大量不确定因素（甲方表结构、WS 消息格式、感应器部署方案等），以下是应对策略：

| 不确定项 | 设计策略 | 变更时的工作量 |
|---------|---------|-------------|
| 甲方的 lot 基础数据表 | LotDataProvider 接口，直接读甲方表 | 写一个实现类 |
| 甲方的 lot 固定工序表 | ProcessDataProvider 接口，直接读甲方表 | 写一个实现类 |
| 甲方的品名额外工序表 | ProcessDataProvider 接口，直接读甲方表 | 写一个实现类 |
| WS 消息格式变化 | WsSensorEventAdapter 负责解析，输出 UnifiedFlowEvent | 改一个适配器 |
| 新增事件源（扫码枪等） | 实现新的 FlowEventAdapter | 加一个适配器类 |
| 测定棚架 WS 格式 | sensor_process_mapping 统一映射，适配器转换格式 | 改映射数据 + 适配器 |
| 异常信息文案 | 可配置的消息模板 | 改配置 |
| 绑定感应器位置变更 | 修改 sensor_process_mapping 中的 sensor_type | 改数据库记录 |
| 新工序判定方式 | 实现新的 FlowStrategy | 加一个策略类 |
| 水墨屏 API 格式 | IDisplayService 接口预留，实现类对接 | 改一个实现类 |
